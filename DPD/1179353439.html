Overview

DevOps Pipeline & Deployment is an initiative to automate activities and practices around DLX development. We want to enable more thorough testing, avoid deploying breaking changes, and move from file-attachment to web-hosted deployment for our DLX projects.

Projects

Development Pipeline

The development pipeline runs whenever a developer finishes work on a ticket and submits a Pull Request to ask that their work be evaluated and approved for merge to the main code line for their project. This pipeline needs to execute automated tests for the project and make the report available to reviewers so they can quickly reject requests that would introduce new bugs to the overall project.

Deployment Pipeline

The deployment pipeline runs whenever a developer merges their approved ticket to the main code branch. This pipeline needs to run the same tests as the development pipeline, but will not produce a report. Instead the compiled code is uploaded to a web host for users to access.

DLX Web Hosting / LTI Integration

Our final DevOps step is hosting the built DLX applications. This could be done in two modes: unauthenticated hosting for casual testing purposes, and authenticated hosting for testing learning tool access (via link configured in Moodle) or as a production Digital Learning Experienced (via link configured in eConestoga and exposed in a learnerâ€™s course shell.

Services

Bitbucket

Bitbucket is our source control repository. It hosts many projects, some of which are set up to initiate a pipeline on our build server during events like a Pull Request or Merge.

Jenkins

Jenkins runs several build pipelines (on varlabjenkins.eastus.cloudapp.azure.com) which are invoked in different cases by webhook calls from Bitbucket. One is the development build, which runs tests and produces a report. The other is the deployment build, which runs tests and if everything works perfectly uploads the resulting build (Unity WebGL) to our hosting server.

DLX-WebHost

Our deployment pipeline (as well as some manual processes) deploy builds to dlx-webhost.canadacentral.cloudapp.azure.com which we often prefer to redirect to with https through webdlx.vconestoga.com

Moodle

Moodle is an open-source Learning Management System (LMS) and we use it to test hosting / deployment of DLX before making it available to the production system eConestoga. VARLab hosts its own instance on https://vconestoga.duckdns.org/

For web-hosted builds, these have to be set up so that they authenticate through the Learning Tool Interoperability (LTI) interface, effectively requiring a link and authorized token from Moodle.

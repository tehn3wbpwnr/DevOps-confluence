Prerequisites

In order to set up the DevOps pipeline on your Unity project, you will need:

  • Tests for your project (ideally both PlayMode and EditMode tests)

  • The Code Coverage package installed (refer to the documentation on setting up Code Coverage)

Jenkins Set-Up

This section requires a Jenkins account with admin privileges.

Creating a Pipeline Project

Log into the Jenkins dashboard. In the menu on the left-hand side, click “New Item.”

Enter the name of your project. Then select “Folder” and click “OK.” We need to separate our pipeline project from the others using folders, so that later on, each project can have their own Bitbucket API credentials.

The configuration page will open once you click “OK.” It’s fine to leave all these fields blank. You are free to define a display name or description for your folder if you’d like.

On the folder screen, click “Create a job.”

We will be adding our pipeline job to this folder. Enter the name of your project, suffixed with “Pipeline.” Again, this name should have no spaces. Then, click “Pipeline” and “OK.”

Project Settings

On the configuration page, the description is optional but useful. Underneath the description, check “Do not allow concurrent builds.”

The pipeline is parameterized, but to speed up the onboarding process and make setting up a Pipeline job easier, both the parameters and Generic Webhook Trigger variables have been specified in the pipeline scripts.

After the Generic Webhook Trigger plugin is set up for our pipeline job, we will manually run the pipeline once. This initial build will always fail, but will auto-populate the parameters and variables from the script.

Generic Webhook Trigger

Under the next section, “Build Triggers,” select only “Generic Webhook Trigger.” We’ll configure this plugin to automatically populate our parameters with payload data from Bitbucket.

Eventually, we need to give the Generic Webhook Trigger plugin a token. This token should be unique. It will be used by the plugin to identify the repository that the webhook is associated with. Without this, the pipeline will fail. We will do this later on, when we configure the webhook in Bitbucket.

Pipeline Settings

Under the “Pipeline” section, change the “Definition” dropdown to “Pipeline script from SCM.” This means that Jenkins will retrieve the pipeline script from a project repository, so we can avoid defining the script in the Jenkins dashboard.

Set the “SCM” dropdown to “Git.” Enter the DevOps Pipeline repository’s git SSH URL, which can be found on the repository page when you click the “Clone” button.

Set the credentials to “varlab-machine-user (Bitbucket CI-Catherine ED25519). CI-Catherine is the VARLab’s machine user profile, and we’ll later be giving her repository permissions in order for her to access your project and pull requests.

Under the “Branches to build” section, we want to specify the main branch. Type this as “/*main”.

The rest of the SCM settings can be left as their default values.

Your project is now set up on Jenkins. Click “Save.”

Deployment Pipeline Set-Up

Repeat the steps above to set up another pipeline under the same project folder as your other pipeline. Ideally, give it a name suffixed with “Deployment” to differentiate the job from the pull-request pipeline.

Lastly, when setting up the pipeline settings, change the script path to “JenkinsfileDeployment.”

Bitbucket Set-Up

This section requires admin privileges on your project’s repository.

Default Reviewers

Continuous Integration Catherine (our machine user) should be added to pull requests as a default reviewer. Click “Default reviewers” in the left menu, then click “Add reviewers” to add the machine user.

Webhook

Navigate to your project’s repository on Bitbucket and click “Repository Settings” in the left toolbar, then “Repository permissions.” Change the Automation group’s permissions to Admin.

Next, click “Webhooks” in the left toolbar. Add a webhook by clicking the “Add webhook” button.

Name your webhook something unique and identifiable. The format of the URL should be: http://jenkins.varlab.org/generic-webhook-trigger/invoke?token=YOUR_TOKEN_HERE, with the token being your choice. We’ll be using this unique token for the Generic Webhook Trigger’s settings. We can leave the secret blank.

Make sure the “Active” checkbox is checked, otherwise the webhook won’t work.

Set your webhook triggers in the next section. These triggers will be what triggers the pipeline to build, so at minimum, “Created”, “Updated” and “Merged” need to be checked under the “Pull Request” column. You are free to add additional triggers as needed for your project. Click “Save” when you are finished.

Next, navigate back to the Jenkins dashboard and click on your project folder. In the left menu, click “Credentials.”

You should see a section on the credentials page called “Stores scoped to <FOLDER_NAME>.” There is only one domain, the (global) domain. Any credentials entered in this domain can be accessed by all children of the folder. Hover over (global) and click the arrow next to it, then click “Add credentials.”

Your new credentials are going to be of type “Secret text,” which you can select from the “Kind” dropdown menu. Copy the token you created for the Bitbucket webhook into the “Secret” textbox.

It’s important to set your credentials' ID to “trigger-token.” This is what identifies your credentials to the pipeline script. You can set your description to whatever you prefer. When you’re done, click “Create.”

API Access Token

Since workspace-wide API tokens are a paid service on Bitbucket, we unfortuntately need to create a new access token for each pipeline project. We’ll be setting up our access token on Jenkins as well.

Navigate to “Access Tokens” in the left menu, then click “Create Repository Access Token.”

Give your access token the following scopes:

  • Admin permissions for repositories

  • Read and Write permissions for pull requests

When you create the access token, be sure to copy it before closing the window. For security reasons, Bitbucket will only show you this token once.

Next, navigate back to the Jenkins dashboard and click on your project folder. Follow the same steps as with the webhook set-up to create a new secret text credential. Copy your access token into the “Secret” text box.

It’s important to set your credentials' ID to “bitbucket-access-token.” This is what identifies your credentials to the pipeline script. You can set your description to whatever you prefer. When you’re done, click “Create.”

Finally, return to your project folder. For each pipeline, click on them and then select “Build Now.” This first initial build will always fail, but will auto generate the parameters and variables that were mentioned earlier.

Unity Set-Up

The WebGL Play Mode Tests stage is currently nonfunctional because of a Unity bug. Until a fix is found, the pipeline will not be running this stage. This section is skippable for now but will be kept in for future reference.

Since projects need to be deployed to eConestoga, the pipeline builds and runs playmode tests with WebGL. However, there is a bug in Unity that will sometimes use the wrong graphics API when running these tasks through the command line. If the graphics API it tries to use is not compatible with WebGL, it will try to fall back on WebGL 1, which is deprecated and incompatibile with linear color space.

To fix this, we need to add a copy of a correct EditorUserBuildSettings file to the repo. This file is located in the Library folder of your Unity project, which should not be tracked by source control. To avoid tracking this folder, we’ll copy the file into another folder called “ci” that we will push to the remote repo.

Open your Unity project and navigate to File > Build Settings. Select WebGL and, leaving everything else at its default value, click “Switch Platform.”

Once it finishes, we should also make sure the Player Settings are correct. Click the Player Settings button in the lower left corner. The window should automatically open to WebGL’s player settings.

Click the “Other Settings” dropdown. If “Auto Graphics API” is checked, uncheck it. If you can see WebGL 1 in your list of Graphics APIs, select it and click the minus sign to remove it. Your only graphics API should be WebGL 2.

Close this window and close Unity. Open your Unity project in File Explorer. Create a folder named “ci” in the root of your project, if it does not already exist.

Open the Library folder and locate the EditorUserBuildSettings.asset file. Copy (don’t move!) this file into your project’s ci folder.

Push all of these changes to your remote repository. The pipeline will take care of the rest of the work, moving the EditorUserBuildSettings.asset file into it’s own local Library folder before attempting to build.

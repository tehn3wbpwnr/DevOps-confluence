1. Overview of SonarQube

SonarQube prodivdes reports on bugs, security recommendations, code standards, duplicated code, and code coverage for unit tests based on static analysis.

2. Installation

To run the SonarQube server on our jenkins-win-docker, both the SonarQube database and SonarQube server must be installed. https://docs.sonarsource.com/sonarqube/10.4/setup-and-upgrade/install-the-server/introduction/

To perform static anylsis, we need to use Sonar Scanner. Sonar Scanner is a client tool that works with SonarQube to execute the actual code analysis.

2.1 Install SonarQube database (PostgreSQL)

SonarQube supports three different external database engines – Microsoft SQL Server, Oracle, and PostgreSQL. https://docs.sonarsource.com/sonarqube/10.4/setup-and-upgrade/install-the-server/installing-the-database/

The DevOps team is currently using locally installed PostgreSQL, because it has simpler requirments and allows easy data access via its UI.

1. Install PostgreSQL

Follow these steps easily by visiting: https://www.youtube.com/watch?v=-bPepNFduUg

2. Create a user for SonarQube in PostgreSQL

Click Login/Group Roles → Create → Login/Group Roles

Add a user name and password.

This credentials will be used in the sonar.properties file.

Assign superuser privileges to the user to ensure that SonarQube can access any database server.

References:

Initial setup in PostgreSQL for SonarQube: https://www.youtube.com/watch?v=47v5zk1EugI&t=1080s

2.2 Install SonarQube server (Docker image)

SonarQube server is installed using a Docker image to isoltae dependencies.

Instread of specifying the JDK version (JDK 17 or 21), we use a Docker image to simplicity.

1. Prepare sonar.properties File

The sonar.properties file is the configuration file for SonarQube. Set up the database connection PostgreSQL by adding:

sonar.jdbc.username=<postgreSQL user>
sonar.jdbc.password=<postgreSQL password>
sonar.jdbc.url=jdbc:postgresql://host.docker.internal:5432/sonarqube

Use the PostgreSQL credentials created during step “2.1 Install SonarQube database (PostgreSQL)”.

The sonar.properties file will be mounted to the SonarQube Docker container. This approach ensures that the SonarQube configuration is secure and easy to modify

2. Prepare .yml File for Docker-Compose

Using a .yml file, we can easily set up the SonarQube server and ensure database connection.

Current location of the .yml file used in jenkins-win-docker:
C:\Users\ci-catherine\Desktop\SonarQube\docker-local-DB-compose.yml

version: "3"

services:
  sonarqube:
    image: sonarqube
    ports:
      - "9000:9000"
    restart: always   # Ensures the server restarts automatically
    volumes:
      - "C:/Users/ci-catherine/Desktop/SonarQube/sonar.properties:/opt/sonarqube/conf/sonar.properties"
      # Replace with local file
      
      - sonarqube_data:/opt/sonarqube/data
      - sonarqube_extensions:/opt/sonarqube/extensions
      - sonarqube_logs:/opt/sonarqube/logs
      

volumes:
  sonarqube_conf:
  sonarqube_data:
  sonarqube_extensions:
  sonarqube_logs:
  # Mount volumes to retain information across container restarts

3. Run SonarQube Server Container

a. Open Command Prompt.

b. Navigate to the .yml file directory C:\Users\ci-catherine\Desktop\SonarQube

c. Run the following command: docker-compose -f <docker-compose-name> up -d

About Running Docker Container Instance

Basically, we can create a Docker container instance using docker run <name_of_docker_image> with default configurations.
However, docker-compose -f <docker-compose-name> up -d allows us to create and manage Docker container instances with user-defined configurations specified in the .yml file.

4. Access SonarQube

SonarQube runs on port 9000.

Login credentials are stored in:
C:\Users\ci-catherine\Desktop\SonarQube\IDnPW.txt

https://docs.sonarsource.com/sonarqube/latest/setup-and-upgrade/install-the-server/installing-sonarqube-from-docker/

2.3 Install Sonar Scanner Plugin in Jenkins

1. Generate SonarQube Tokens

Navigate to SonarQube instacne in the virtual machine and generate essential tokens to authenticate Jenkins with SonarQube.

a. Generate a User Token
Navigate to My Account → Security → Generate Tokens
Fill out the following:

  • Name: <User token name> (Choose any descriptive value)

  • Type: User Token

  • Expires in: Set as needed

b. Create a Project and Generate a Project Token
Navigate to Projects → Create Project → Local project
Fill out the following:

  • Project display name: <Project display name> (Choose any descriptive value)

  • Project key: Automatically generated as <Project display name> is set.

Select “Use the global setting“ option and click “Create project“

Select “Locally” for Analysis Method

Fill out the following:

  • Token name: <Project token name> (Choose any descriptive value)

  • Expires in: Set as needed

Ensure to store token value secuerly, as they cannot be viewd again after generation.

2. Install SonarQube Scanner Plugin

Navigate to Jenkins → Manage Jenkins → Plugins → Available plugins

Search for “sonar scanner“ and install the plugin.

3. Register SonarQube User Token to Jenkins Credentials

Navigate to Jenkins → Manage Jenkins → Credentials → Global Credentials

Click “Add Credentials“ and fill out the following:

  • Kind: Secret text

  • Secret: <SonarQube user token value>

  • ID: <Credential ID> (Choose any descriptive value)

4. Configure SonarQube Server in Jenkins

Navigate to Jenkins → Manage Jenkins → Systems → SonarQube servers

Configure the following fields:

  • Environment variables: Check

  • Name: <SonarQube server name> (Choose any descriptive value)

  • Server URL: http://localhost:9000/sonarqube

Note: This is set as <localhost>/sonarqube because HTTPS access will be configured in Step: 4.-Proxy-Configuration-for-HTTPS

  • Server authentication token: <SonarQube credential> (from Step 3 above)

5. Install Sonar Scanner in Jenkins

Navigate to Jenkins → Manage Jenkins → Tools → SonarQube Scanner installations

Click “Add SonarQube Scanner“ and configure the following:

  • Name: <SonarQube scanner name> (Choose any descriptive value)

7. Apply SonarQube in Jenkins Scripts

Add Static Analysis stage in Jenkinsfile as shown below. This will execute static analysis using SonarQube.

3. Project Settings on the SonarQube Instance

1. Set Up File Exclusions

Navigate to Proejct → Project Settings → General Settings → Analysis Scope

Add excluded paths for scopes as needed to exclude specific files or directories from analysis.

2. Configure Paths for Code Coverage Files

Navigate to Proejct → Project Settings → General Settings → Languages

Select “JavaScript / TypeScript“.

Scroll down to “Tests and Coverage”, and add the paths for the lcov.info file.

What is the 'lcov.info' file?

lcov.info is a file that stores code coverage information. This file is written in LCOV format and is mainly used to record code coverage data after test execution in JavaScript, TypeScript, and other language projects.

lcov.info contains the execution status for each line of the specified test files (DA), Test Name (NT), and Source File path (SF). So, SonarQube can visualize the test results as CodeCoverage based on this information.

For example, Jenkins runs tests using Jest, which generates an lcov.info file as a test result. Jest saves the lcov.info file in the specified directory. Finally, SonarQube reads this lcov.info file and visualizes the code coverage data.

3. Create Quality Gates

  • What are Quality Gates?
    Quality Gates are predefined quality standards that must be met before a project can proceed to the next phase.

Example Criteria of Quality Gates

  • Bug: New code should be bug-free

  • Vulnerabilities: There should be no security vulnerabilities

  • Code Smells: There should be no code that is difficult to maintain

  • Coverage: Test coverage should be above a certain percentage
    >> e.g. Coverage should be above 80%.

  • Duplicated Lines: Code duplication should not exceed an acceptable threshold
    >> e.g. Duplication rate should be below 3%.

Here is the documentation like for the SonarQube Quality Gate:

https://docs.sonarsource.com/sonarqube-server/latest/instance-administration/analysis-functions/quality-gates/

Navigate to Quality Gates from the top navigation bar.

Create a customized quality gate to adjust quality standards according to the project’s requirements.

4. Proxy Configuration for HTTPS

To enable HTTPS access for SonarQube and restrict access to specific sources, configure the proxy settings for both Nginx and SonarQube.

Since the Jenkins UI is already hosted on the root URL, a separate URL must be configured to host SonarQube for public access.

4.1 Nginx Configuration

Modify the Nginx configuration file:
C:\Program Files\Nginx\conf\nginx.conf

server {
        listen       443 ssl;
        server_name  jenkins.vconestoga.com;
        
        //...
        
          location /sonarqube/ {
              if ($http_referer = ""){
                  return 403;
              }
              if ($http_referer !~* "(https://bitbucket.org|https://jenkins.vconestoga.com)"){
                  return 403;
              }
              # The above code restricts access to requests from outside Bitbucket or not from internal sources
              
              proxy_pass http://localhost:9000/sonarqube/;
              proxy_set_header Host $host;
              proxy_set_header X-Real-IP $remote_addr;
              proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
              proxy_set_header X-Forwarded-Proto $scheme;
          }
        
        }

4.2 SonarQube Configuration

Modify the sonar.properties file:
C:\Users\ci-catherine\Desktop\SonarQube\sonar.properties

Set the sonar.web.context to /sonarqube (this can be different) to match the proxy configuration.

sonar.web.context=/sonarqube

Restart the container if it's already running:

docker restart <container_id>

After these changes, SonarQube will be accessible at http://localhost:9000/sonarqube, instead of http://localhost:9000 inside jenkins-win-docker.

In the public, SonarQube will be accessibile at https://jenkins.vconestoga.com/sonarqube, available exclusively through the Bitbucket link.

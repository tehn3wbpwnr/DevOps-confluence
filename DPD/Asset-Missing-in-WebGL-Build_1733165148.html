DevOps / Pipeline & Deployment : Asset Missing in WebGL Build

Created by Jindo Kim, last modified on Apr 10, 2025

This is a copy of PR: https://bitbucket.org/VARLab/devops-linux-jenkins/pull-requests/41

This document covers the following topics:

  • Git Large File Issue (Missing Art Model in WebGL Build)

    • Analysis and resolution of the issue where some Art Models were missing during WebGL builds.

    • Optimization of the Jenkins Pipeline Script to execute git lfs pull only when necessary by improving its execution conditions.

1. Overview (Missing Art Model)

  • While using Jenkins to build the Tattoo Project for WebGL, we encountered an issue where some art models were missing intermittently without a clear pattern. At the time of discovery, we temporarily resolved the issue by running git lfs pull on the Jenkins server, though the root cause was not fully identified.

  • In this solution, we identified a method to check for files that Git LFS is not tracking by using the command git lfs ls-files | awk '$2 == "-" {print $3}'. Based on this, we improved the Jenkins Pipeline Script to execute git lfs pull only when such untracked files are detected. (Previously, git lfs pull was always executed in the Pipeline Script, which led to unknown issues.)

2. Symptoms

  1. Recurrence After Temporary Fixes

    1. Although running git lfs pull temporarily resolves the issue, assets are still missing intermittently during WebGL builds at certain times.

  2. Issues with Running git lfs pull in the Pipeline

    1. When git lfs pull was configured to run every time within the Jenkins Pipeline, unexpected errors occurred.

      1. As a result, there is a need for a method to validate and execute git lfs pull only when necessary.

      2. At the time of the issue, error logs were not saved, making it difficult to pinpoint the exact cause. However, the problem is suspected to be related to Git source control.

3. Root Cause Analysis

Upon investigation, it was confirmed that the issue of certain art models being missing during WebGL builds stems from a case sensitivity issue caused by differences between Windows and Linux environments:

  • The .gitattributes file, which manages Git Large File Storage (LFS), failed to recognize.FBX files (with uppercase extensions) or encountered unexpected issues, causing Git LFS to not track these files properly.

  • In Linux environments, where case sensitivity is enforced, even if .fbx is included in the .gitattributes file, .FBX files with uppercase extensions are not recognized unless explicitly specified.

  • An attempt was made to resolve this by adding the configuration *.FBX filter=lfs diff=lfs merge=lfs -text to the .gitattributes file. However, Git did not recognize this change at all.

  • To verify how Git was managing large files in the project, the git lfs ls-files command was executed.

    • Additionally, no clear documentation was found regarding the exact meaning of the - and * symbols in the output of the git lfs ls-files command. Further research is needed on this point. However, it was confirmed that files marked with the - symbol corresponding to the art models were missing in the WebGL build.

  • Running git lfs ls-files --debug revealed that the checkout: false status could occur due to various reasons. In this specific case, since the issue occurred only in the Linux environment (Jenkins) and not in the Windows environment, it is highly likely that the root cause lies in case sensitivity issues and improper Git LFS configuration

4. Attempted Solutions

The following solutions were attempted, but they did not fully resolve the issue:

  1. Git Configuration Change (Failed)

    • An attempt was made to address the case sensitivity issue by applying the setting:

      • git config core.ignoreCase true However, this did not have the desired effect.

  2. Git Attributes Configuration and Manual Modification (Failed)

    • The .gitattributes file was updated to explicitly recognize.FBX extensions with Git LFS. The following steps were performed:

      1. Run ``git rm --cached <filename>`` to remove the files from the index.

      2. Execute ``git reset <filename>`` to restore the files.

      3. Run ``git add .`` to stage the changes.

      4. Commit the changes with git commit -m "Fix LFS issue".

    • Despite these efforts, the issue persisted, and the solution failed to resolve the problem.

5. Implemented Solution

(The following explanation is based on the results of manual steps, and the same logic has been applied to the Pipeline Script.)

  1. Verification and Re-registration of Git LFS Files

    • The git lfs ls-files command is used to verify the list of files tracked by Git LFS and check for any missing files.

  2. Improved Execution Condition for git lfs pull in the Pipeline

    • Instead of running git lfs pull unconditionally, a validation method is implemented to execute it only when necessary.

      • The command git lfs ls-files | awk '$2 == "-" {print $3}' is utilized to identify files that Git LFS is not tracking.

      • Based on this, the Jenkins Pipeline Script was updated to execute git lfs pull only when such untracked files are detected.

    • This saves the detailed execution logs of git lfs pull into the lfs_pull.log file for further analysis.

    • After running git lfs pull, no problematic files were found in the output of git lfs ls-files. As shown in the WebGL screenshot below, titled "New WebGL Build after 'git lfs pull' Command,” all art models were successfully included in the build.

6.1. Understanding the Current Jenkins Logic and Why the Issue Persisted Together.

  • The root cause of the recurring issue lies in the reset operations (git reset, git restore .) within the pipeline. These commands undo the changes made by git lfs pull, reverting the repository to its original state.

  • Prepare Workspace Stage:

    • Files managed by Git LFS were checked for issues using commands like git lfs ls-files.

    • Problematic files (not tracked by Git LFS) were identified.

  • Execution of git lfs pull:

    • After running git lfs pull, the problematic files were no longer detected, as they were successfully downloaded and tracked by Git LFS.

    • The WebGL build proceeded successfully in this state, producing correct results.

  • Reset Operations in Later Stages:

    • In subsequent stages (e.g., Workprepare Stage or Post Stage), commands like 'git reset' and 'git restore .' were executed to clean up or reset the workspace.

    • These operations reverted the repository to its original state, including the issues with Git LFS tracking.

  • Pipeline Cycle Completion:

    • By the end of the pipeline cycle, the repository returned to its initial problematic state where Git LFS did not track certain files.

    • As a result, the same issue reoccurred in subsequent pipeline runs unless git lfs pull was executed again.

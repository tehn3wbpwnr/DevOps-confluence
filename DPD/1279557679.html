About Build Behaviors(Strategy) Setting

: There are a lot of behaviors we can use in Jenkins, but this document will be focus on only two things.
One is the strategy of how to discover the branches in a repository.
Another is the strategy of how to build and test the project.

1. Discover branches

: This strategy allows us to sort the branches in the selected repository. So, we can build some certain branches that crossponding with this strategy.

Currently, the most appropriate option will be ‘Only branches that are also filed as PRs'. If you select this option, the 'Multi-Branch Source Plugin’ will be discover the branches that are created a pull request in given Bitbucket repository like the image below.

However, you can select 'All branches' option if you want to manage all the branches regardless of PR creation then the Jenkins will discover and display all branches in a repository.

Not recommend selecting 'Exclude branches that are also filed as PRs' because it will ignore the PR request branch which is our target project.

2. Discover pull requests from origin

: This behavior determines how PRs (Pull Requests) are explored and processed in Jenkins' Multi-Branch Pipeline settings.

Therefore, considering the server's processing capacity and build strategy, it is reasonable to choose one of the three options below.

  • Option 1. Merging the pull request with the current target branch revision

: This option discovers the PR as if it has already been merged with the target branch.

>> The build and test process occurs using the state after the PR is merged into the target branch, helping preemptively resolve any merge conflicts or integration issues.

  • Option 2. The current pull request revision (Default)

: This option discovers the PR and checks out the latest revision of the corresponding branch.

 

  • Option 3. Both the current pull request revision and the pull request merged with the current target branch revision (Not Recommended)

: This option discovers both branches which are merged with the target branch and the latest revision of the corresponding branch. Both projects will be built at the same time.
>> It will trigger two builds in one event, so it will take too much resources in Jenkins server and could break our system.

Introduction

Welcome to the DevOps Team! This page is here to help ease your transition into a new team, There will be definitions and explanations of all the new technology you will exploring through your journey at VAR Lab.

First, let’s start with the basics: What is DevOps?

Simply put, DevOps is the intersection of development and IT operations. We help bridge the gap between the code developers create and fully deployed, production-ready applications. This includes ensuring applications are tested, deployed, and monitored effectively, often through the use of automation tools. Below, we’ll dive into some of the key terms and technologies you'll encounter in your day-to-day work, we will keep everything at a high level without digging too deep into details on this page, that can come later.

Definitions

The tech industry is full of jargon that can feel overwhelming at first, like saying “We automate our CI-CD pipeline, to ensure rapid reliable deployments.” What does this mean? Lets break it down.

CI/CD Pipeline: This stands for continuous integration, and continuous deployment. You can think of it as an assembly line for code, where each step along the way, actions are taken on the code (Such as testing), automatically.

Automation: There is lots of talk about automation, lets define what we mean. Essentially we want to leverage tools, such as running tests, or building code. Which are repetitive tasks to do by hand, and have it so these tasks are performed automatically.

Pipeline: This is how we automate things. Think of it as a piece of code or script that runs when something happens like a Pull Request is made.

Linting: We have mentioned tests, but there is also a thing called linting. Linting is a process of analyzing code to ensure it meets a certain standard. As an example we have a style guide where variables are supposed to be named in a certain case based on public, private, etc. We can create a rule set and scan the code to make sure it follows the style guide keeping a consistent well maintained code base!

Virtual Machine: A Virtual Machine (VM) is a software-based emulation of a physical computer, we use Azure to host our VMs in the cloud, we have both windows and Linux based VMs in use.

What Does Our Pipeline Look Like?

Before going over too many definitions lets look at our pipeline. We use Bitbucket to host our remote code repositories. A developer will work on a ticket for a feature than create a pull request. This is the beginning of where DevOps comes in! So what happens then?

  1. Automatically when a Pull Request(PR) is made it will then send an http request to our pipeline server! This is what triggers the pipeline to run, with associated information such as what repository, branch, etc.

  2. The pipeline begins running its associated script for the type of project (Unity or JavaScript).

    1. Pipeline uses git to pull the code from the PR, and performs a local merge with main to check for conflicts.

    2. Linting! Then we scan the code against our style guide. This being Non-critical is allowed to have errors, and we provide a report back to the developers letting them know the files and lines of code that broke the rules.

      1. For some projects there could also be a static code analysis step here that checks the project for known security risks, etc.

    3. Tests are then run, this could happen in multiple stages depending on type of project. (Unity has Edit Mode and Play Mode tests). Based on the results we also provide some sort of feedback to the developers, via a report of some sort.

    4. We build the project. This ensures the code compiles okay, and may have other steps such as deploying the finished product (A WebGL build for Unity, or Docker container for JavaScript.

    5. Lastly the pipeline then informs the developers of the pipeline result, and sends any remaining reports, such as test results, code scanning results, etc.

  3. When the pipeline is finished it may have passed the built code on to another server, which may do extra things such as deploying a project so there may be additional steps afterwards, but we won’t worry about those right now.

And that is what a pipeline looks like at a high level! NOTE: This keeps the explanation simple, however there can be additional context based on how the pipeline runs such as, is this PR still open? (Then it won’t deploy), or if the PR was merged (It will deploy the build). This is just to give you a high level break down of what the pipeline does, later we will cover it in more depth.

Additional Tools

The DevOps team doesn’t just manage pipelines, we also have some additional things that we work on. Such as managing Virtual Machines that get finished builds deployed to. A major one we have is our LTI Web Server. Another acronym?! Lets cover it.

What is LTI? LTI stands for “Learning Tools Interoperability”, Which is a fancy way of saying its a standard used for educational tools (such as eConestoga) to communicate with external programs/tools.

What does it do? Our learning simulations will be hosted on this web server, the web server uses this standard to communicate with eConestoga to ensure users are authorized and authenticated, as well as provide information to simulations like what is the student’s name, unique identifier, etc. It can also be used to submit grades back to eConestoga!

So our pipeline will take the finished Unity build, and transfer it onto the Virtual Machine running our web server. Then it can be accessed on eConestoga without having to manually upload any files!

DevOps / Pipeline & Deployment : Summer 2024 DevOps Term Recap

Created by Joshua Moore, last modified on Aug 15, 2024

Accomplished This Term

This was the first real term of having a full dedicated DevOps team, filled with a lot of learning. This section will cover some of the things we learned about, explored, and implemented this term.

Jenkins

Initial half of summer term was focused on Jenkins, learning about the initial pipeline implementation and how it functioned, then improving and expanding upon it. This could be split into 3 categories General Jenkins improvements, the PR Created/updated pipeline, and the PR Merged pipeline.

General Jenkins additions this term

  • switched from cloning repos for every new PR to a single shared repo managed similarly to how you manage git when working on your local machine, contextually switching branches to match your current needs.

  • added “BuildServer” tag to running of tests to allow exclusion of problematic tests when there is no Graphics interface.

  • so many bug fixes

PR Creation/merge pipeline summary

  • Handles git swapping of branches, ensuring local repo is reset then matches expected remote.

  • Runs Edit-mode and Play-mode tests for the Unity project

  • Based off tests will report back to Bitbucket via REST API with number of failed tests and percentage of code coverage as this pipeline is informative.

  • Changed build options to increase speed and efficiency since this pipeline is only concerned with compilation errors.

PR Merge pipeline summary

  • Handles git pulling/resetting of main

  • Runs Edit-mode and Play-mode tests for the Unity project

  • No sending of test reports as this pipelines purpose is for deployment

  • Builds the project then sends it to our dlx-webhost server and calls a shell script to deploy it to the active webserver.

LTI Webserver

The second half of the term was spent learning about a research project that was built by the vConestoga team. They were investigating the use of LTI to host our WebGL builds on a remote server and hook them up to LMS(eConestoga/Moodle) as external tools. There was the foundations of a web server put together using the LTIjs library and we took those foundations and moved to it a proof of concept using Moodle and putting together a Unity package that can be used as part of our CORE framework to allow sims to interact with the LTI server.

LTI Summary

  • Webserver hosted on linux VM, internally routed to via https tunnel on NGINX server

  • built using express for the back end using the LTIjs library

  • uses Oauth/OpenID for Authentication/Authorization

  • Hosts the sims WebGL builds in public folder, sets redirect routes to those folders following authentication

  • Created CORE LTI package hosted on our upm registry to allow sims to interface with the LTI server retrieving learner information

  • so many bug fixes

  • Added code to automatically update the server code when pr is merged to target main

  • CRON job setup to check pm2 status and restart server if there is an issue pinging its address

DevOps / Pipeline & Deployment : DevOps Pipeline Features

Created by Leo Bunting (Unlicensed), last modified on Dec 04, 2023

If you followed the DevOps pipeline onboarding instructions, the pipeline should build and test your project if you have either created a pull request or updated an existing pull request. Changing when the pipeline runs only requires making changes to the webhook triggers that were set up here.

At various stages in the pipeline, your pull request will be updated with the following information.

Currently, the pipeline can only build one branch at a time. If a new build is triggered while one is already running, it will be put in a queue and will run when it is its turn. This can cause long waits for build results. There are plans to add multi-branch functionality to the pipeline in the future.

Build Status

Each stage of the pipeline’s lifecycle is reported to the corresponding Bitbucket pull request. Jenkins will update the build status when the pipeline is in progress, and when it has passed, failed, or been aborted.

The build status can be found in a pull request’s right-side menu, under the tab titled “X out of Y builds failed/passed.” Each build is labeled with its corresponding build ID on Jenkins.

Test Results

The pipeline runs both a project’s EditMode and PlayMode tests. PlayMode tests are run in both the Editor and a WebGL build. The Editor PlayMode tests are only used for code coverage calculation, and only the WebGL test results will be displayed on a pull request.

PlayMode tests are run in WebGL to ensure we catch any bugs or failing tests that might only show up when running the WebGL build. This stage of the pipeline also simultaneously builds the project for WebGL.

Test results can be found in a pull request’s right-side menu, under the Reports tab. They’ll be titled with their corresponding build number and type of test.

Code Coverage Results

The pipeline calculates your branch’s line coverage using the EditMode and (editor) PlayMode code coverage results. Currently only the line coverage is reported back to the pull request.

Code coverage results can be found in a pull request’s right-side menu, under the Reports tab. It will be titled with its corresponding build number.

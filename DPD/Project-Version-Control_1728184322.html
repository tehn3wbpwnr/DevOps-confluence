DevOps / Pipeline & Deployment : Project Version Control

Created by Chris Park, last modified on Apr 02, 2025

Currently, the deployment pipeline in Jenkins manages the patch version number instead of the web developers. This document describes how Jenkins handles patch versioning and container image management.

Here is a summary of the deployment process on Jenkins:

  • Manage the patch version

  • Build Docker images

  • Push Docker images to Azure Container Registry

  • Delete outdated images from Azure Container Registry

  • Update the Azure Container App

1. About versions.json

: The main purpose of this configuration file is to track patch numbers by Jenkins itself. However, versions.json includes not only patch numbers.

Sample versions.json format
{
  "server_patch_version": 0,
  "client_patch_version": 0,
  "latest_server_version": "1.0.0",
  "latest_client_version": "1.0.0",
  "latest_ticket_number" : ""
}

1.2 Where versions.json format can be found

: versions.json configuration format is defined at the

Managed Jenkins >> Managed File >> Config File Management >> versions.json

The configuration ID is required when the Jenkins brings the versions.json format in the deployment build process.

1.3 How Jenkins creates versions.json during the pipeline build?

:  Jenkins checks whether JsJenkins/versions.json exists or not before it reads a file.

  • If versions.json is not found in {env.WORKSPACE}/JsJenkins:
    - A new versions.json file is created by calling createVersionJsonFile().
    - The newly created file is then initialized using initializeVersionJsonFile().
    - All version-related data is stored as a Map and shared across the entire pipeline build.

  • If versions.json already exist in {env.WORKSPACE}/JsJenkins
    - The existing file is read, and its contents are stored as a Map and shared across the entire pipeline build.

We can call versions.json file format using ‘configFileProvider' with Configuration ID and file name.

configFileProvider([configFile(fileId: 'configuration_id_here', targetLocation: 'versions.json')]) {

We can’t just read the configuration format using configFileProvider. It only allows us to CREAT a configuration file based on the format

1.4 When versions.json is updated?

:  Each server and client patch number is locally incremented at the end of the ‘Check Build and Deploy Condition’ stage, after all deployment conditions have been verified.

At this point, the updated patch numbers are not yet written to versions.json.

The updateVersionsJson() function is called at the end of the pipeline only if the build completes with a ‘SUCCESSFUL’ status. This function updates the patch numbers, full version numbers, and the associated ticket number in versions.json.

If either the server or client deployment stage fails, versions.json will remain unchanged.

 

2. ACR repository management

 : Jenkins JS deployment pipeline remotely deletes outdated container images from ACR repository by calling deleteOutdatedAcrImages(). The detailed explanation logic is described at the next content with example Scenario.

2.1 Version Update Scenario 

Before Update

In general, ACR stores:

  • All latest major versions of container images as backups

  • One minor version of the container image for the in-progress major version

  • One patch version of the container image for the in-progress major version

After Major Version Update

When the major version is updated:

  • Minor and patch version of images of the previous major version are deleted

After Patch Version Update

When a patch version is updated:

  • The previous patch version of image is stored as a backup

After Another Patch Version Update

When another patch version is updated:

  • The previous patch version of image is deleted

  • The previously latest image is stored as a patch version backup

After Minor Version Update

When minor version is updated:

  • The previously latest image is stored as a patch version backup

  • previous patch version of image is stored as a minor version backup

3. Deployment Report

The result of deployment is delivered by Bitbucket Build Status API. Each deployed client and server deployed version number will be passed if all server and client project are successfully pushed to ACR and updated on ACA.

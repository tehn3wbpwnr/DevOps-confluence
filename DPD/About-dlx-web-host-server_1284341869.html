About dlx-web host server

: dlx-web host server is the deployment server for dlx projects. This server has all the latest version of dlx builds provided by the Jenkins deployment pipeline.

The main purpose of this server is host the latest version of dlx WebGL builds to LMS(eConestoga or Moodle) when it requests a certain project.

So, LTIjs service is running through the pm2 process manager and listening a HTTP:REQUEST from LMS to host the right project.

1. Auto Script(Triggered by the Jenkins deployment pipeline)

: This shell script will be triggered by the Jenkins deployment pipeline when it copys and pasts the WebGL builds into web-dlx host server.

This script has two purposes.

  • Copy and paste the WebGL build “/var/www/html/{project_directory}“ to “~/DLX-Server/devopsdlxserver/public/{project_directory}“.
    >> It is because, we want to keep the backup at “/var/www/html/{project_directory}“ directory, but actually the WebGL builds should be placed at “public/{project_directory}“ directory to be hosted by LTIjs service.

  • Update urls(TemplateData, Build, SCORM script) in ‘index.html'
    >> 'index.html’ is created during the WebGL build, but the original urls work with LTIjs service. So, we have to update them.
    >> Just simply add “/{project_directory}/“ in front of all urls

This shell script is located at “~/ShellScripts/UpdateBuildURL.sh”

2. pm2

: A process manager for Node.js applications, designed to help you easily manage and monitor applications in a production environment. PM2 plays a critical role in maintaining the availability and performance of applications, and provides features such as clustering, load balancing, and automatic restart.

Currently, two services are running through pm2 like the image below

Each service can be run based on the 'ecosystem.config.js' and this configuration.

Path of each 'ecosystem.config.js'

: ~/DLX-Server/devopsdlxserver/ecosystem.config.js
>> Configuration file path of LTI js Server Updater

Useful pm2 command to manage services

Start the application
: pm2 start ecosystem.config.js

Check status of application
: pm2 status

Monitoring the applications
: pm2 monit

Check a certain application’s log
: pm2 log {process_id} or {application_name}

3.1 Dynamic Registration

: This process is the process of registering a new platform (LMS) and granting access to external tools through Open_ID. In short, it is a method that supports the process of registering an external tool in LMS.

This method will be launched when LMS requests HTTP:POST with '/register' parameter.
>> This process happens in “Add LTI Advantage“ which is the process of external tool registration.
public key, private key, platform, and platform data are created in MongoDB through this process.

3.2 Deep Linking Service

: This is a proccess that allows LMS to create and manage direct links to specific learning content via URLs after checking whether LMS have access to the resource.

This service happens when an instructor selects the one of the content in the selection list.
>> This process happens when an instructor clicks “Select content“ button in content editing page and submits a certain content.

3.3 Names and Roles

: The Names and Roles Provisioning Service (NRPS) is a service for sharing user information and roles between an LMS and external tools. This service allows educational tools to securely obtain information about a user's name, email address, and the role the user has in a course (e.g., student, teacher, administrator, etc.).

NRPS can get data using LTIjs' NamesAndRoles.getMembers() method with ltik parameter.

3.4 Grading Service

: This service is used to exchange learner performance data between the LMS and external learning tools. Through this service, students' grades can be updated remotely when certain conditions are met.

ActivityProgress
: Represents the progress of a learner in completing a specific activity.

GradingProgress
: Represents the progress of grading. It represents the steps from when a learner submits an assignment to when the assignment is graded.

Score & MaximumScore
: Represents the score the learner received (Score) and the maximum score they can receive for the activity (MaximumScore).

4. MongoDB

: MongoDB is the default database that LTIjs supports. Currently, dlx-web host server has MongoDB 7.0.12 and the LTIjs service is connected with ‘ltiManager’ Account.

We can check the the MongoDB connection url in the '.env' file at the “~/DLX-Server/devopsdlxserver/.env“ directory.

[Account Information]

Currently, MongoDB has two accounts {1. ciCatherine} and {2. ltiManager}.
>> ‘ciCatherine’ account has full access to the admin collection and is suitable for creating new accounts.
>> ‘ltiManager’ account has a read&write access to ‘test’ and ‘DLX' collections.

[Collection Information]

Currently, MongoDB has three active collections

>> ‘test’ collection is the data set for LTI authentications.
>> ‘DLX’ collection is the data set for dlx projects and course lists.
>> 'admin’ collection is the data set to store the accounts data such as ciCatherine or ltiManager.

[How to connect to MongoDB command shell]
: dlx-web host server has a 2.2.14 mongosh(MongoDB Shell) and we can connect to this command shell using one of the account explained above.

Example Command
{ mongosh $MONGO_URI --username account_here --password password_here --authentication Database admin }

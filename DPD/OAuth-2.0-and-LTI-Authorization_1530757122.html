This documentation introduces the OAuth 2.0 authorization protocol, which serves as the foundation of OpenID Connect (OIDC), and its application in our LTI system.

1. What is OAuth 2.0?

: OAuth stands for “Open Authorization“ and it designed to allow a website or application to access resources hosted by other web apps on behalf of a user.

For example, OAuth 2.0 is normally used in the login process through extensions like OpenID Connect (OIDC), allowing users to sign in to different websites using a single account, such as a Google or Microsoft account.

In our case, Conestoga students are authenticated when they log in to the eConestoga webpage. Afterward, the DLX server verifies the credentials of both the student and the LMS (eConestoga) before providing access to resources.

2. Principles of OAuth 2.0

The main principle of OAuth 2.0 is an Access Token. Access Token is a piece of data that represents the authorization to access resources on behalf of the end-user.

OAuth 2.0 doesn’t define a specific format for Access Tokens. However, in some contexts, the JSON Web Token (JWT) format is often used.

Here is the JWT format specification for LTI
: https://www.imsglobal.org/spec/lti/v1p3

In our LTI Advantage also use JWT form of access token called ‘ltik'. This token doesn’t have expiration date and passed as a cookie to client.

3. OAuth2.0 Roles

The idea of roles is part of the core specification of the OAuth2.0 authorization framework. These define the essential components of an OAuth 2.0 system, and are as follows:

  • Resource Owner (Student)
    : The user or system that owns the protected resources and can grant access to them.

  • Client (Student Browser)
    : The client is the system that requires access to the protected resources. To access resources, the Client must hold the appropriate Access Token.

  • Authorization Server (DLX Server - External Tool Provider)
    : This server receives requests from the Client for Access Tokens and issues them upon successful authentication and consent by the Resource Owner. The authorization server exposes two endpoints: the Authorization endpoint, which handles the interactive authentication and consent of the user, and the Token endpoint, which is involved in a machine to machine interaction.

  • Resource Server (eConestoga - LMS)
    : A server that protects the user’s resources and receives access requests from the Client. It accepts and validates an Access Token from the Client and returns the appropriate resources to it.

4. OAuth 2.0 Scopes

: Scopes are an important concept in OAuth 2.0. They are used to specify exactly the reason for which access to resources may be granted. Acceptable scope values, and which resources they relate to, are dependent on the Resource Server.

In our case, the scopes can be found in the id_token, and they define the permissions we have on LMS resources.

5. How Does OAuth 2.0 Work?

At the most basic level, before OAuth 2.0 can be used, the Client must acquire its own credentials, a client_id and client secret, from the Authorization Server in order to identify and authenticate itself when requesting an Access Token.

Using OAuth 2.0, access requests are initiated by the Client, e.g., a mobile app, website, smart TV app, desktop application, etc. The token request, exchange, and response follow this general flow:

  1. The Client requests authorization (authorization request) from the Authorization server, supplying the client id and secret to as identification; it also provides the scopes and an endpoint URI (redirect URI) to send the Access Token or the Authorization Code to.

In our case, the LMS sends a login request to the DLX server at the '/login' endpoint.

An example form is provided at the end of this content.

  1. The Authorization server authenticates the Client and verifies that the requested scopes are permitted.

The DLX server verifies the login request parameters against the data in the local database.

  1. The Resource owner interacts with the Authorization server to grant access.

The grant type we are using is the ‘Implicit Grant’.

  1. The Authorization server redirects back to the Client with either an Authorization Code or Access Token, depending on the grant type, as it will be explained in the next section. A Refresh Token may also be returned.

The DLX server will redirect the client back with the ‘ltik’ JWT access token set as a cookie.

  1. With the Access Token, the Client requests access to the resource from the Resource server.

The client can make API calls to request additional resources to DLX server using the 'ltik' token.

6. Grant Types in OAuth 2.0

In OAuth 2.0, grants are the set of steps a Client has to perform to get resource access authorization.

In our case the scenario matches with...

Implicit Grant: A simplified flow where the Access Token is returned directly to the Client. In the Implicit flow, the authorization server may return the Access Token as a parameter in the callback URI or as a response to a form post.

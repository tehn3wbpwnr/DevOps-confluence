Linting

Reasons and Benefits

  • Aim to maintain consistency within the team by applying standardized coding styles and rules or complexity throughout the development process.

  • https://www.reddit.com/r/devops/comments/yo0y5i/precommit_vs_prepush_vs_cicd_for_linting_and/

    • Can reduce usage and error issues within the CI workflow.

    • Encourages a culture/practice of improving the overall code quality across the team.

Possible Focus Area

Code Formatting

  • Indentation: can enforce indentation rules, detecting mixed tabs and spaces or inconsistent indentation levels.

  • Line length: can set rules to limit the length of code lines.

  • Unused variables: can detect unused variables.

  • Code style consistency: offers various rules to maintain code style consistency, such as the placement of curly braces or the use of parentheses around function arguments.

Syntax Errors:

  • Missing or misplaced parentheses or brackets: can detect incorrect use of parentheses or curly brackets.

  • Undefined variables: can detect variables that have not been defined.

  • Misspelled or undefined functions: can detect misspelled function names or functions that are not defined.

  • Invalid or unexpected characters: can detect abnormal characters within JavaScript code.

Code Standards:

  • Variable naming conventions: can enforce variable naming conventions (e.g., camelCase, snake_case).

  • Code documentation: ESLint does not natively enforce documentation standards, but plugins (e.g., eslint-plugin-jsdoc) can be used to enforce documentation rules like JSDoc.

  • Function complexity: ESLint can detect the complexity of functions and warn if complexity exceeds a certain threshold, but it only supports basic functionality. Therefore, it's advisable to use other tools in Jenkins to perform a more in-depth analysis.

    • https://eslint.org/docs/latest/rules/complexity

    • Linting and static analysis have different scopes for complexity.

  • Code duplication: ESLint, even with extensions, offers very limited support for this feature. Therefore, we might consider using other tools to diagnose code duplication as part of the Jenkins pipeline.

    • Maybe 'jscpd' and other npms can be utilized to cover this area.

    • https://www.npmjs.com/package/jscpd - Package Health Score

    • https://www.npmjs.com/package/postcss-discard-duplicates - Package Health Score

    • Linting and static analysis have different scopes for code duplication.

ESLint Configuration Loading

  • Description: ESLint loads configurations from the .eslintrc file or the package.json file located in the project's root directory. These files define the rules, plugins, and extensions to be applied.

  • Example: You can use the extends option to apply the Airbnb ruleset or configure it to ignore certain file patterns.

File Collection

  • Description: Based on the configured file patterns (e.g., src/\\/\*.js), ESLint collects the files to perform linting. ESLint filters out files according to the ignorePatterns defined in the configuration file.

  • Example: Files or directories specified in the .eslintignore file are ignored, and only the necessary files are collected.

Code Analysis and Rule Application:

  • Description: For each collected file, ESLint parses the code to generate an Abstract Syntax Tree (AST). It then applies the configured rules to this AST to identify code violations.

    • Example: The no-unused-vars rule detects unused variables, or the indent rule identifies incorrect indentation.

  • Issue Reporting and Fixing:

    • Description: Based on the analysis results, ESLint outputs a report of identified issues. ESLint marks rule violations as warnings or errors, and some issues can be automatically fixed.

    • Example: Using the --fix option, ESLint can automatically fix certain issues, such as code formatting or removing unnecessary semicolons.

Result Output:

  • Description: Finally, ESLint outputs the analysis results to the console or saves them to a file, depending on the configuration. The output includes the location and detailed description of each issue found in the files.

    • Example: The stylish format is used to output issues in a readable manner to the console, or the results can be saved as JSON to be used by CI tools.

Commit Restriction:

  • Description: This process must be successfully completed before a commit can be made. This ensures that the code meets the defined standards locally before being committed.

    • Note: As this is part of the pre-commit hook stage, the linting process is executed locally to catch issues early in the development workflow.

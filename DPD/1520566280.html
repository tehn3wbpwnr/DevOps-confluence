DevOps / Pipeline & Deployment : How does the dlx server host the dlx WebGL Build?

Overview

This document describes each “LTI Launch Flow“ for DLX WebGL hosting when students try to play a simulation on eConestoga.

The LTI Launch Flow has two remarkable processes happen during a dlx simulation access.

  1. Open Id Connect(OIDC)

  2. Access to the resource for authorized student

Please note, this is a low level of the “LTI Launch Flow” documentation. Please review the list of documentation below if you don’t know about LTI integration!

>> High Level LTI documentation: All about LTI Integration

1. Open Id Connect(OIDC)

: This is the main authentication and authorization process in LTI integration. OIDC is the protocol that adds user authentication functionality to OAuth 2.0. Namely, the DLX server will check authorization of LMS and student and give authentication token called JSON Web Token(JWT).

OIDC process occurs when the student tries to access to the external tool. Here is the brief steps of OIDC process.

  1. A student click external tool URL(WebGL Build Link) on LMS

  2. LMS sends login request to DLX server

  3. DLX sever confirms LMS’s authorization

  4. DLX server requests “LTI Advantage Login“ which is ODIC to LMS(eConestoga or Moodle)

  5. eConestoga will verify the request of DLX server

  6. Send the id_token to DLX server

  7. Confirm the id_token

  8. DLX server generates ‘ltik’ JWT access token

  9. Redirect to the target endpoint(WebGL Build Link) with 'ltik' access token

This workflow is more detailedly described in this link!

2. Access To Resource

: This process delivers the requested Unity WebGL build to the student's web browser, allowing the student to play a DLX simulation.

During the OIDC authentication, the DLX server receives the “target endpoint URL,” which corresponds to the “LTI Advantage Link URL“ the student clicked to access the DLX Unity WebGL build.

All DLX endpoints must follow the format {dlx_server_url}/dlx/{name_of_dlx_project} (e.g., https://brightspacedlx.vconestoga.com/dlx/rab) so that the route handler can recognize the request and send the correct DLX project back to the client.

Additionally, both the ‘ltik’ access token and the session cookie are required for authentication.

  • If the request is missing the ‘ltik,’ the DLX server will respond with “401: Unauthorized” and the message “NO_LTIK_OR_IDTOKEN_FOUND.”

  • If the request is missing the session cookie, the DLX server will respond with “401: Unauthorized” and the message “Session not found.”

The DLX server will redirect to the target endpoint with all required authentication during the initial request. However, if the Client or WebGL build needs to interact further with the DLX server, they must include the access token and session cookie in subsequent requests.
